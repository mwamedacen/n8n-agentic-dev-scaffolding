# n8n Resync Scripts

## Quick Start

```bash
./resync_workflow.sh dev <workflow_key>           # single workflow
./resync_workflow.sh dev <workflow_key> true      # also strip trigger nodes
./resync_all.sh dev                                # every workflow in dev YAML
```

The Python equivalent (preferred for agents) is `helpers.resync(key, env=None)` and `helpers.dehydrate(json_text)`. The harness's helper surface deliberately omits `resync_all` — agents compose with `for k in list_workflows(): resync(k)`.

## What dehydration does

1. **Fetch** the live workflow JSON via `GET /api/v1/workflows/{id}`.
2. **Strip metadata** that is runtime-specific (see table below).
3. **Restore placeholders** where hydrated values match known patterns:
   - Config values → `{{HYDRATE:env:...}}`
   - File contents → `{{HYDRATE:txt/json/html:...}}`
   - JavaScript inside `// DEHYDRATE:START`/`END` markers → `{{HYDRATE:js:...}}`
   - Trigger UUIDs → `{{HYDRATE:uuid:...}}` (matched against the existing template if present, otherwise auto-generated by node-name slug)
4. **Preserve** nodes, connections, settings, and node positions.

## What gets stripped

| Field | Reason |
|---|---|
| `id` | Per-instance |
| `versionId` | Changes on every save in the UI |
| `updatedAt`, `createdAt` | Volatile |
| `active` | Controlled by deploy scripts |
| `staticData` | Runtime state |
| `pinData` | UI-only test data |
| `tags`, `shared`, `homeProject`, `usedCredentials`, `isArchived` | Per-instance / cloud-only |
| `description`, `activeVersion`, `activeVersionId`, `triggerCount`, `versionCounter`, `scopes`, `parentFolder` | n8n cloud API additions, not authored |
| `meta` | Often empty or per-instance — dropped to keep diff stable |

The list is aligned with `helpers.workflow_semantic_diff`'s ignore set so resync round-trips are byte-stable.

## When to resync

- A teammate edited a workflow in the n8n UI and you need their changes.
- You changed a workflow in the UI for debugging and want to capture the result.
- Periodic backup of live state into version control.

## After resync

1. `git diff n8n/workflows/` — review changes.
2. Re-hydrate locally to verify nothing leaked: `n8n-harness -c "hydrate('<key>')"`.
3. Commit.

## Trigger UUID handling

`dehydrate_template_uuids` runs FIRST so any UUID already mapped to a placeholder in the existing template (by node name) is restored as-is. `dehydrate_trigger_uuids` then auto-generates placeholders for any trigger node that wasn't covered. Net effect: stable templates across resyncs as long as you don't rename trigger nodes.

If you rename a trigger after it's deployed, the next resync will produce a fresh placeholder name (slugified from the new name), and your old name disappears. To keep stable naming, rename in the template first, then deploy.
